# M1.10A/B Guide Information Architecture and Asset Integration Design

Date: 2026-07-05

Status: ready for M1.10A/B implementation

## Scope

M1.10A and M1.10B should be implemented together because the guide information architecture depends on the same spell, talent, icon, tooltip, and link data that the asset layer must provide.

M1.10A covers:

- Static route structure for a GitHub Pages guide site.
- Main guide index and individual spec guide page templates.
- Player-facing copy rules.
- Navigation, filters, metric tooltip definitions, and warning visibility rules.

M1.10B covers:

- Asset cataloging for spell/talent icons and later class/spec media.
- AscensionDB spell hotlinks.
- Static hover tooltip payloads built from normalized entries and DB enrichment.
- Missing-asset fallback behavior.
- Asset and tooltip manifests for auditability.

This design does not implement the CoA-style talent tree renderer from M1.10C, the build diversity selection heuristic from M1.10D, or the role taxonomy overhaul from M1.10E. A/B must create extension points for those milestones without implementing their full behavior.

## Research Summary

GitHub Pages is appropriate for the target output because it serves static HTML, CSS, and JavaScript files from a repository and can optionally use a build process. GitHub documents site and repository size constraints, soft bandwidth limits, build limits, and deployment timeouts, so this project should keep generated assets compact and avoid view-time API dependencies.

Guide reference sites show useful information patterns:

- Icy Veins emphasizes class/spec navigation, clear guide sections, and changelogs.
- Wowhead emphasizes overview pages, quick rules, and spell/item hotlinks.
- Archon emphasizes data context, filters, jump links, stat/talent/gear sections, and alternate builds.
- Method emphasizes compact class guide navigation: introduction, talents, gearing, stats, playstyle/rotation, and related utility sections.

AscensionDB is the appropriate player-facing hotlink target for CoA spell/talent pages. The public site is JavaScript-heavy, and the scraper already uses `https://db.ascension.gg/?spell=<id>&power` for offline tooltip enrichment. Guide pages should link users to `https://db.ascension.gg/?spell=<id>` and use local tooltip payloads so the static site is fast and usable without calling AscensionDB on hover.

## Product Shape

The M1.10A/B output should be a static guide site generated by the existing `python -m coa_meta meta ... --format html` path.

Required output layout:

```text
reports/meta/
  index.html
  meta-report.html
  meta-report.json
  meta-report.md
  assets/
    guide.css
    guide.js
    guide-site-manifest.json
    tooltip-catalog.json
    icons/
      ...
  specs/
    venomancer-stalking.html
    guardian-vanguard.html
    ...
```

`index.html` is the preferred landing page. `meta-report.html` should remain as a compatibility alias for existing tests and user habits.

The static site must work from both a local filesystem preview and a GitHub Pages project path such as `/coa_meta_analyzer/`. All links should be relative. No guide page should require network access for rendering, filtering, hover tooltips, or icons.

## Guide Index

The guide index should answer "which spec guide should I open?" rather than "what is every internal score value?".

Sections:

- Hero/header: project name, generated timestamp, builder/capture version, and a short theorycraft disclaimer.
- Filters: class, role, encounter, confidence, and warnings-present.
- Class/spec grid: one card per spec result.
- Data notes: report assumptions, artifact provenance, and warnings.

Spec cards:

- Class name and spec name.
- Role label and role provenance when available.
- Spec icon/media if available; otherwise a styled class/spec monogram.
- Best build label from `summary` or the top build.
- Confidence badge.
- Warning badge only when warnings exist.
- "Open guide" link to the spec page.

Metrics such as Projected DPS Index and Confidence should be visible only as compact chips or within tooltips. The card should avoid exposing score breakdowns, selected node lists, and implementation-specific text.

## Spec Guide Page

Each spec guide should use a consistent navigation model so players can move through the page like a familiar class guide.

Required sections:

1. Overview
2. Recommended Builds
3. Talents
4. Rotation
5. Stats
6. Weapons and Armor
7. Abilities and Talents
8. Warnings, hidden when empty
9. Data Notes

M1.10A/B should implement the section shell, navigation, copy rules, ability/talent list, icon/link/tooltip behavior, and warnings visibility. M1.10C will replace the initial Talents placeholder with a builder-like tree. M1.10D will replace the initial Recommended Builds and Rotation summaries with distinct playstyle cards and core loops.

Overview:

- Start with the class/spec name and role.
- Use player-facing language: "Best for", "Watch out for", "Core idea", "Data confidence".
- Avoid phrases such as "heuristic scorer", "feature vector", or "beam search".
- Link technical metric chips to local tooltip definitions.

Recommended Builds:

- Show the current top builds, but label this as "Early theorycraft picks" until M1.10D changes selection.
- Each build card should show rank, confidence, role, selected active ability highlights, and a link to the Talents section.
- Do not list every selected node inline on first render; reserve the full list for expandable details.

Talents:

- In A/B, provide a structured placeholder and ability/talent list filtered to the build/spec.
- Include a clear note that the interactive tree arrives in M1.10C.
- Preserve fields needed by C in the guide model: `row`, `col`, `connected_node_ids`, `required_ids`, `max_rank`, `ae_cost`, `te_cost`, `required_tab_ae`, `required_tab_te`, `required_level`, and availability.

Rotation:

- Keep the current generated APL summary in a player-facing wrapper.
- Hide internal action category names when possible.
- M1.10D will replace this with core-loop extraction.

Stats:

- Show one yellow warning panel per spec: "Stat priorities are early theorycraft until simulations or combat logs are available."
- Render stat rows as compact ordered chips.

Weapons and Armor:

- Split into "Best targets for this spec" and "Available to this class/spec".
- In A/B, the section may still show low-confidence placeholders from the current gear recommendation.
- Missing item/icon data should be explicit but concise.

Abilities and Talents:

- Show a searchable or filterable list of relevant class and spec nodes.
- Each row/card must include icon, name, type, source tree, required level, cost, tags, and a tooltip-enabled DB link when spell ID exists.
- Passive and active abilities should be visually distinct.

Warnings:

- Do not render the section when report, spec, build, or gear warnings are empty.
- When warnings exist, translate common warnings into player-facing copy first and include the raw warning code in a collapsed technical detail.

Data Notes:

- Show generated timestamp, builder name/slug, input artifact paths, report schema version, and source assumptions.

## Data Model Boundary

The guide site should be rendered from a derived guide model. The renderer should not inspect arbitrary report dicts throughout HTML string templates.

Proposed package files:

- `coa_meta/guide_models.py`: dataclasses for guide site, guide spec, guide build card, guide node, guide asset, guide tooltip, and metric definitions.
- `coa_meta/guide_builder.py`: joins `MetaReport`, `TalentRepository`, optional class metadata, optional DB tooltip rows, and optional asset catalog into the guide model.
- `coa_meta/guide_assets.py`: asset catalog and resolver.
- `coa_meta/guide_tooltips.py`: tooltip catalog and DB link helpers.
- `coa_meta/guide_rendering.py`: static HTML/CSS/JS rendering.
- `coa_meta/guide_writer.py`: writes the site output layout.

`coa_meta.reporting` should keep report generation and legacy wrapper functions, but bulky HTML should move out of that file.

## Asset Catalog

Asset source priority:

1. Local generated/copied guide asset under `reports/meta/assets/`.
2. Local scraper asset under `--asset-root`.
3. Normalized or DB icon identifier converted into a stable placeholder.
4. Styled text fallback.

The asset catalog should normalize icon names by lowercasing and removing non-alphanumeric characters, matching current `AssetResolver` behavior. It should also preserve source provenance.

Guide asset record:

```json
{
  "schema_version": "coa-guide-asset-v1",
  "asset_id": "icon:interface-icons-shared-strike",
  "kind": "icon",
  "source": "asset_root",
  "source_path": "coa_scraper/data/raw/...",
  "output_path": "assets/icons/shared-strike.png",
  "label": "Shared Strike",
  "missing": false
}
```

M1.10B should not block on perfect icon capture. It should create deterministic placeholders when icon files are unavailable so the layout can be designed and tested.

## Tooltip Catalog

Tooltip source priority:

1. Matched AscensionDB `coa_db_spell_tooltips.jsonl` row.
2. Normalized `description_html` and `description_text`.
3. Plain fallback using node name, type, costs, level, and source tree.

Tooltip payload:

```json
{
  "schema_version": "coa-guide-tooltip-v1",
  "tooltip_id": "spell:2001",
  "entry_id": 201,
  "spell_id": 2001,
  "name": "Damage Talent",
  "html": "<strong>Damage Talent</strong><p>Increases damage over time.</p>",
  "text": "Damage Talent. Increases damage over time.",
  "db_url": "https://db.ascension.gg/?spell=2001",
  "source": "normalized",
  "source_confidence": "medium",
  "warnings": []
}
```

Tooltips must be sanitized before writing HTML or JSON. The first implementation should allow a small, local HTML subset: `br`, `span`, `strong`, `em`, `small`, `p`, and `div`, with all event handler attributes removed. If sanitization is not implemented in the first task, tooltip HTML should be escaped and rendered as text.

## JavaScript Boundary

M1.10A/B JavaScript should be small and progressive:

- Filter cards on the guide index.
- Toggle expandable build/node details.
- Render local tooltip popovers from `tooltip-catalog.json`.
- Persist no user data.
- Make no network requests.

The site should still be readable without JavaScript. Filters and hover popovers can degrade to visible links and inline summaries.

## Styling Direction

The visual language should be fel/void without copying guide sites:

- Background: deep near-black violet.
- Primary accent: fel green.
- Secondary accent: void purple.
- Status accent: amber for warnings.
- Use glow, border, and texture effects sparingly around cards, active nav, and icon frames.
- Keep content width generous: main content should prefer 1180-1320px max width, with readable line lengths inside text panels.
- Avoid the current squished table layout.

A/B should create the CSS tokens, layout primitives, card styles, tooltip style, warning style, and icon frame style. Later milestones can add stronger tree-specific visuals.

## Testing Strategy

Unit tests:

- Build a guide model from `tests/fixtures/meta_report_fixture.jsonl`.
- Verify index/spec routes and relative links.
- Verify warnings sections are omitted when empty and present when warnings exist.
- Verify metric definitions exist for Projected DPS Index and Confidence.
- Verify spell IDs produce AscensionDB hotlinks.
- Verify tooltip fallback uses normalized text when DB data is absent.
- Verify DB tooltip rows override normalized tooltip text when matched.
- Verify missing icons produce deterministic placeholder records.

Snapshot-like tests:

- Render the index and one spec page.
- Assert player-facing section headings are present.
- Assert implementation terms like "beam search" are absent.
- Assert `index.html`, `meta-report.html`, `assets/guide.css`, `assets/guide.js`, and `assets/tooltip-catalog.json` are written.

Manual smoke:

```bash
python -m coa_meta meta \
  --entries coa_scraper/dist/coa_entries.jsonl \
  --classes coa_scraper/dist/coa_classes.json \
  --out reports/meta \
  --format html \
  --class Venomancer \
  --spec Stalking \
  --top 1
```

Open `reports/meta/index.html` locally and verify the page loads, spec link opens, filters work, tooltips appear, and no remote network calls are required for the guide shell.

## Open Decisions For Later Milestones

- Exact class/spec hero art source. A/B should support media records but use placeholders when source art is unavailable.
- Exact AscensionDB page URL format for non-spell entities such as items, buffs, and effects. A/B should only guarantee spell links.
- Whether to use generated bitmap class/spec art. This can be decided after the static layout exists.
- Whether the final frontend becomes a separate static-site bundle. A/B should keep generation inside Python until the site complexity justifies a dedicated frontend build.
